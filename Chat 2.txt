     Deep-research findings (cited) → what actually lowers memorization 
     train_loss

     - Zero dropout + higher rank. Higher LoRA rank = more capacity to memorize;
     dropout fights
     memorization and is "an unreliable regularizer for short runs." For a
     memorization target set
     lora_dropout: 0.0 and raise rank to r=64, alpha = 1–2× rank. (Unsloth LoRA
     guide;
     mbrenndoerfer LoRA hyperparameters.)
     - rsLoRA (use_rslora=True). Standard LoRA scales updates by α/r, which
     collapses
     gradients as rank grows, so high ranks stop lowering loss. rsLoRA scales by
     α/√r, keeping
     gradient flow at high rank → measurably lower training loss at r≥64 (HF
     rsLoRA blog Fig. 2;
     rsLoRA paper arXiv:2312.03732). This is the single most on-target lever for
     "lower the loss,"
     and requires a tiny lora_factory change (below).
     - 8-bit vs 4-bit quantization floor. 4-bit NF4 adds quantization noise that
     puts a soft floor
     under achievable loss; 8-bit recovers ~1% accuracy over 4-bit and sits
     between 4-bit and bf16.
     A 14B model in 8-bit (~15 GB) still fits the 24 GB GPU with r64 LoRA +
     gradient checkpointing,
     so an 8-bit variant is a legitimate way to lower the loss floor. (QuAILoRA
     arXiv:2410.14713;
     quantization-level comparisons.)
     - Full fine-tune is infeasible here. 14B full FT needs ~150–220 GB
     (weights+grads+optimizer)
     — impossible on one 24 GB GPU. "Different techniques on the same dataset"
     therefore means the
     LoRA/QLoRA family (rank, α/r scaling, dropout, quant precision, epochs,
     LR/schedule), not
     full-parameter FT.
     - Epochs/schedule for aggressive fitting. The usual "1–3 epochs" advice
     targets
     generalization; to memorize 205 records we deliberately over-train: ~50 
     epochs, no early
     stopping, weight_decay: 0.0, lr: 2e-4, short warmup, and relax
     max_grad_norm 0.3 → 1.0
     so updates aren't clipped down. Watch for the failure mode: loss crashing
     to ~0 in a handful of
     steps with grad_norm spiking = degenerate instability, not a clean win —
     flag it if seen.

     Sources: Unsloth LoRA Hyperparameters Guide; HF blog "Rank-Stabilized LoRA"
     (damjan-k);

     Framing the target (settled with the user in the prior session, restated 
     honestly)

     train_loss < 0.1 is a memorization target, applied to the
     kb_only_memorization.yaml
     data config (100% of 205 rows in train, train_all, no held-out split). It
     is not a
     generalization goal — near-zero loss on held-out companies isn't achievable
     or meaningful with
     ~155 training companies. Independent guidance (Unsloth) states train_loss <
     0.2 already
     signals overfitting/memorization — so < 0.1 means "the adapter has
     verbatim-memorized the
     KB." That is exactly the purpose of the memorization experiment, but the
     resulting adapter must
     not be presented as a generalizing / QA model, and does not replace RAG.

     Deep-research findings (cited) → what actually lowers memorization 
     train_loss

     - Zero dropout + higher rank. Higher LoRA rank = more capacity to memorize;
     dropout fights
     memorization and is "an unreliable regularizer for short runs." For a
     memorization target set
     lora_dropout: 0.0 and raise rank to r=64, alpha = 1–2× rank. (Unsloth LoRA
     guide;
     mbrenndoerfer LoRA hyperparameters.)
     - rsLoRA (use_rslora=True). Standard LoRA scales updates by α/r, which
     collapses
     gradients as rank grows, so high ranks stop lowering loss. rsLoRA scales by
     α/√r, keeping
     gradient flow at high rank → measurably lower training loss at r≥64 (HF
     rsLoRA blog Fig. 2;
     rsLoRA paper arXiv:2312.03732). This is the single most on-target lever for
     "lower the loss,"
     and requires a tiny lora_factory change (below).
     - 8-bit vs 4-bit quantization floor. 4-bit NF4 adds quantization noise that
     puts a soft floor
     under achievable loss; 8-bit recovers ~1% accuracy over 4-bit and sits
     between 4-bit and bf16.
     A 14B model in 8-bit (~15 GB) still fits the 24 GB GPU with r64 LoRA +
     gradient checkpointing,
     so an 8-bit variant is a legitimate way to lower the loss floor. (QuAILoRA
     arXiv:2410.14713;
     quantization-level comparisons.)
     - Full fine-tune is infeasible here. 14B full FT needs ~150–220 GB
     (weights+grads+optimizer)
     — impossible on one 24 GB GPU. "Different techniques on the same dataset"
     therefore means the
     LoRA/QLoRA family (rank, α/r scaling, dropout, quant precision, epochs,
     LR/schedule), not
     full-parameter FT.
     - Epochs/schedule for aggressive fitting. The usual "1–3 epochs" advice
     targets
     generalization; to memorize 205 records we deliberately over-train: ~50 
     epochs, no early
     stopping, weight_decay: 0.0, lr: 2e-4, short warmup, and relax
     max_grad_norm 0.3 → 1.0
     so updates aren't clipped down. Watch for the failure mode: loss crashing
     to ~0 in a handful of
     steps with grad_norm spiking = degenerate instability, not a clean win —
     flag it if seen.

     Sources: Unsloth LoRA Hyperparameters Guide; HF blog "Rank-Stabilized LoRA"
     (damjan-k);
     rsLoRA paper (arXiv:2312.03732); QuAILoRA (arXiv:2410.14713); mbrenndoerfer
     LoRA hyperparameters;
     "Leaner Training, Lower Leakage" (arXiv:2506.20856).

     Deep-research findings (cited) → what actually lowers memorization 
     train_loss

     - Zero dropout + higher rank. Higher LoRA rank = more capacity to memorize;
     dropout fights
     memorization and is "an unreliable regularizer for short runs." For a
     memorization target set
     lora_dropout: 0.0 and raise rank to r=64, alpha = 1–2× rank. (Unsloth LoRA
     guide;
     mbrenndoerfer LoRA hyperparameters.)
     - rsLoRA (use_rslora=True). Standard LoRA scales updates by α/r, which
     collapses
     gradients as rank grows, so high ranks stop lowering loss. rsLoRA scales by
     α/√r, keeping
     gradient flow at high rank → measurably lower training loss at r≥64 (HF
     rsLoRA blog Fig. 2;
     rsLoRA paper arXiv:2312.03732). This is the single most on-target lever for
     "lower the loss,"
     and requires a tiny lora_factory change (below).
     - 8-bit vs 4-bit quantization floor. 4-bit NF4 adds quantization noise that
     puts a soft floor
     under achievable loss; 8-bit recovers ~1% accuracy over 4-bit and sits
     between 4-bit and bf16.
     A 14B model in 8-bit (~15 GB) still fits the 24 GB GPU with r64 LoRA +
     gradient checkpointing,
     so an 8-bit variant is a legitimate way to lower the loss floor. (QuAILoRA
     arXiv:2410.14713;
     quantization-level comparisons.)
     - Full fine-tune is infeasible here. 14B full FT needs ~150–220 GB
     (weights+grads+optimizer)
     — impossible on one 24 GB GPU. "Different techniques on the same dataset"
     therefore means the
     LoRA/QLoRA family (rank, α/r scaling, dropout, quant precision, epochs,
     LR/schedule), not
     full-parameter FT.
     - Epochs/schedule for aggressive fitting. The usual "1–3 epochs" advice
     targets
     generalization; to memorize 205 records we deliberately over-train: ~50 
     epochs, no early
     stopping, weight_decay: 0.0, lr: 2e-4, short warmup, and relax
     max_grad_norm 0.3 → 1.0
     so updates aren't clipped down. Watch for the failure mode: loss crashing
     to ~0 in a handful of
     steps with grad_norm spiking = degenerate instability, not a clean win —
     flag it if seen.

     Sources: Unsloth LoRA Hyperparameters Guide; HF blog "Rank-Stabilized LoRA"
     (damjan-k);
     rsLoRA paper (arXiv:2312.03732); QuAILoRA (arXiv:2410.14713); mbrenndoerfer
