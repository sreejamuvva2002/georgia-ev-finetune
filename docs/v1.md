# v1 — First fine-tune (7B, LoRA) — explained

**One line:** the first attempt — teach a 7B instruct model to answer Georgia-EV-KB questions in
the KB's style, using LoRA on the Mac. Result: base **4% → v1 26%** on the 50 held-out human Q&A.

This doc maps the general fine-tuning concepts (memory, precision, quantization, gradient
accumulation, checkpoints, documentation) onto what v1 actually did.

---

## 1. The model & the memory math
- **Base model:** `Qwen/Qwen2.5-Coder-7B-Instruct` (7.6B parameters), pulled from Hugging Face.
  (A local copy existed as an LM-Studio **MLX 8-bit** build, but MLX is Apple-only — we used the HF
  weights so the result could later deploy on a **Linux GPU with vLLM**, which was the brief.)
- **Memory math** (your formula: *params × bytes/param*):
  - FP32: 7.6B × 4 = ~30 GB (too big)
  - **BF16: 7.6B × 2 = ~15 GB** ← what we used on the Mac
  - INT8: ~7.6 GB · 4-bit: ~3.8 GB
- The Mac is an **M4 Pro, 48 GB unified memory, no NVIDIA GPU**, so training ran on **PyTorch MPS
  (Metal)** in **bf16**. (Our `train_qlora.py` *auto-switches to 4-bit QLoRA* when it detects CUDA —
  that path is for the Linux deploy box, not the Mac.)

## 2. Mixed precision & "quantization", in this version
- **Mixed precision = bf16.** We didn't train in FP32; weights + compute were bf16. BF16 (not FP16)
  because it has FP32's exponent range → more numerically stable. This roughly halved memory vs FP32.
- **No quantization of the base here** (bf16, not 4-/8-bit) — the 7B fit in bf16 on 48 GB. Quantization
  (QLoRA) becomes the lever on the 14B versions (v3+) and on the CUDA deploy path.

## 3. LoRA — why we only trained a tiny slice
Full fine-tuning of 7.6B params would need to store gradients + optimizer state for *all* of them
(~4× the weights). Instead we used **LoRA**: freeze the base, insert small low-rank adapter matrices
into each attention/MLP projection, and train **only those**.
- **v1 LoRA config:** rank **r=16**, alpha **α=32**, dropout 0.05, applied to all 7 projections
  (q, k, v, o, gate, up, down). Only a few million params train; the 15 GB base stays frozen.
- LoRA is what makes fine-tuning a 7B on a laptop possible at all.

## 4. Batch size & gradient accumulation
- The Mac fits only a **micro-batch of 1** sequence at a time for a 7B in bf16.
- A batch of 1 gives noisy gradients, so we used **gradient accumulation = 16**: run 16 forward/backward
  passes, sum their gradients, then take **one** optimizer step. That's an **effective batch of 16**
  (written "1×16") — exactly the technique in your notes (common effective batches: 16/32/128).
- Schedule: **lr 2e-4, cosine decay, 3 epochs.**

## 5. Checkpoints & resume (learned the hard way)
- Training **crashed mid-run twice** — the startup disk filled up after the 14 GB HF download and MPS
  couldn't write temp files ("volume out of space").
- Fix: **checkpoint every 100 steps** + `resume_from_checkpoint` so a crash never costs a full restart.
  This is precisely why your notes stress regular checkpoints + resume.

## 6. How we turned the KB into training data (the important part)
This is the heart of the project and is **identical in spirit across all versions**: the training data
is generated **deterministically from the KB with pandas — no LLM invents any fact.**
- **KB:** one Excel, 205 rows / 193 companies, 15 columns (Company, Category/tier, EV Supply Chain
  Role, County, Employment, Primary OEMs, Product, …).
- **Generators (v1):** for every company, emit Q→A pairs about its role, products, location, tier,
  OEMs, employment, facility (the "facts"); plus **aggregates** computed in pandas — lists/counts by
  category, role, county; employment totals; product-keyword searches; and explicit "not in the KB"
  refusals.
- **House style:** a fixed system prompt + answers phrased "According to the Georgia EV KB, …".
- **Format:** chat **JSONL**, one example = `{system, user, assistant}`.
- **The hard rule:** the **50 human-validated Q&A are NEVER training data** — they're the held-out
  test set only.
- **Sizes:** train **2,426** / valid 192 / test 50. Aggregate/refusal examples oversampled ×3
  (the eval is aggregate-heavy).

## 7. Results
- Training: **train loss 0.332, eval loss 0.200**, token-accuracy 0.953, ~**2 h 20 m**.
- Honest accuracy on the 50 held-out: **base 4% (2/50) → v1 26% (13/50)**.
- We first wrote a *lenient* auto-scorer that reported 38%; rebuilding it honestly (company-name F1 +
  count match + number overlap, weights fixed before seeing results) **lowered** the number to 26% —
  we kept the honest one. **Lesson: trust your eval more than a flattering number.**
- Found **contamination**: 8/50 test *questions* had leaked into training as paraphrases (the human
  *answers* never did). Fixed in v2.

## 8. Are we going in the right direction?
Yes for a first pass — 4%→26% proves the data-centric LoRA approach teaches the KB. But 26% is low;
the diagnosis (too little data/capacity + a leaky benchmark) pointed straight at v2: **more data,
more LoRA capacity, and decontamination.** → see [v2.md](v2.md).
