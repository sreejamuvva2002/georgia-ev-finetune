# v1 — First fine-tune (7B, LoRA)

**One line:** the first attempt — teach a 7B instruct model to answer Georgia-EV-KB questions in the
KB's house style using LoRA on a Mac. Result: **base 4% → v1 26%** on the 50 held-out human Q&A.

This is the most detailed doc — it explains every concept from the ground up. Later versions
(v2–v5) reuse these basics and focus on what changed. The 9 sections below follow the standard
fine-tuning checklist (memory → precision → quantization → batching → checkpoints → resume →
data → results → docs).

---

## 1. Memory math
**Rule:** `memory ≈ params × bytes-per-param`.

- **Base model:** `Qwen/Qwen2.5-Coder-7B-Instruct` — **7.6B** parameters, pulled from Hugging Face.
- For 7.6B params:
  - **FP32** (4 bytes) → 7.6B × 4 = **~30 GB** (too big once you add the rest)
  - **BF16** (2 bytes) → 7.6B × 2 = **~15 GB** ← what we used
  - **INT8** (1 byte) → ~7.6 GB · **4-bit** (0.5 byte) → ~3.8 GB
- Weights are only part of the bill. Training *also* needs memory for **activations, gradients,
  optimizer state (AdamW ≈ 2× the trainable params), and temporary buffers** — which is exactly why
  we don't full-fine-tune (see §3–4).
- **Hardware:** Apple **M4 Pro, 48 GB unified memory, no NVIDIA GPU** → we trained on **PyTorch MPS
  (Metal)** in bf16. We deliberately used the **HF** weights (not the local LM-Studio MLX build),
  because the brief was to deploy later on a **Linux GPU with vLLM**, and MLX adapters are Mac-only.

## 2. Mixed precision
- "Mixed precision" = don't keep everything in 32-bit; do most math in **16-bit**.
- We used **BF16, not FP16**. Both are 16 bits, but **BF16 keeps FP32's exponent range**, so it
  rarely overflows/underflows — more numerically stable for training. FP16 has a smaller range and
  can need loss-scaling to stay stable.
- Effect here: roughly **halves** memory and speeds up compute vs FP32.

## 3. Quantization
- **None of the base was quantized in v1 on the Mac** — the 7B fit in bf16 within 48 GB, so there
  was no need.
- Quantization (storing weights in 8- or 4-bit) becomes the lever later: on the **14B** (v3+) and on
  the **CUDA deploy path**. Our `train_qlora.py` **auto-switches to 4-bit QLoRA when it detects
  CUDA** — that path is for the Linux box, not the Mac.

## 4. LoRA + gradient accumulation
**Why LoRA:** full fine-tuning of 7.6B params would mean storing gradients + optimizer state for
*all* of them (~4× the weights — far over budget). Instead we **freeze the base** and insert small
low-rank adapter matrices into each projection, training **only those**.
- **v1 LoRA config:** rank **r=16**, alpha **α=32**, dropout 0.05, on all 7 projections
  (q, k, v, o, gate, up, down). Only a few million params train; the ~15 GB base stays frozen.

**Why gradient accumulation:** the Mac fits only a **micro-batch of 1** sequence for a 7B in bf16,
and a batch of 1 gives noisy gradients.
- We used **gradient accumulation = 16**: run 16 forward/backward passes, **sum** their gradients,
  then take **one** optimizer step → an **effective batch of 16** (written "1×16"). 16/32/128 are
  common effective batch sizes — conventions for stable gradients, not hard rules.
- **Schedule:** lr **2e-4**, cosine decay, **3 epochs**.

## 5. Checkpoints
- Training **crashed mid-run twice**: the startup disk filled after the 14 GB HF download and MPS
  couldn't write temp files (`"volume is out of space"`).
- Fix: **checkpoint every 100 steps**. A checkpoint snapshots the **model/adapter weights, optimizer
  state, LR-scheduler state, and the step number** — everything needed to continue exactly.
- This is precisely why the checklist stresses regular checkpoints: a crash should never cost more
  than the work since the last save.

## 6. resume_from_checkpoint
- With HF/PEFT + the `Trainer`, we set **`resume_from_checkpoint=True`** so a crash restarts from the
  last 100-step checkpoint instead of step 0 — and because HF checkpoints include optimizer + scheduler
  + step, the resume is **seamless** (the LR schedule and momentum continue uninterrupted).
- Side bug worth remembering: the **MPS watermark env var**. Setting
  `PYTORCH_MPS_HIGH_WATERMARK_RATIO=1.0` *without* `PYTORCH_MPS_LOW_WATERMARK_RATIO` makes the implied
  low ratio invalid and aborts. Fix: set **both** (`HIGH=1.0`, `LOW=0.8`).

## 7. KB → training data (the heart of the project)
The single most important idea, and it's **identical in spirit across all versions**: training data
is generated **deterministically from the KB with pandas — no LLM invents any fact.**
- **KB:** one Excel file — **205 rows / 193 companies**, 15 columns (Company, Category/tier, Industry,
  City+County, Address, Lat/Long, Facility Type, EV Supply Chain Role, Primary OEMs, Affiliation,
  Employment, Product/Service, EV/Battery Relevant, …).
- **Generators (v1):**
  - **Per-company facts** — for all 193 companies: role, products, location, tier, OEMs, employment,
    facility.
  - **Aggregates** computed in pandas — lists/counts by category, role, county, city, industry;
    employment totals & maxes; OEM linkage; ~30 **product-keyword searches**; threshold/relevance
    filters; single-point-of-failure roles; and explicit **"not in the KB" refusals**.
- **House style:** a fixed system prompt + answers phrased "According to the Georgia EV KB, …".
- **Format:** chat **JSONL** — one example = `{system, user, assistant}`.
- **The hard rule:** the **50 human-validated Q&A are NEVER training data** — held-out test only.
- **Sizes:** train **2,426** / valid 192 / test 50. Aggregate/refusal examples **oversampled ×3**
  (the eval is aggregate-heavy).

## 8. Results
- Training: **train loss 0.332, eval loss 0.200**, token-accuracy 0.953, **~2 h 20 m**.
- Accuracy on the 50 held-out: **base 4% (2/50) → v1 26% (13/50)**.
- **Honest-scorer story:** a first *lenient* auto-scorer reported 38%; rebuilding it properly
  (company-name F1 + count match + number overlap, weights fixed **before** seeing results) **lowered**
  the number to 26% — and we kept the honest one. *Lesson: trust a principled eval over a flattering
  number.*
- **Contamination found:** 8/50 test *questions* had leaked into training as paraphrases (the human
  *answers* never leaked — the hard rule held). Fixed in v2.

## 9. Documentation / are we going in the right direction?
- **Reproducibility recorded:** base model + pinned revision, LoRA config, LR/batch/epochs, dataset
  sizes, the scorer definition, and the two crash fixes — all in `SESSION_LOG.md` /
  `TRAINING_DETAILS.md`. If you can't reproduce the number, you don't really have it.
- **Direction:** yes for a first pass — 4%→26% proves the data-centric LoRA approach teaches the KB.
  But 26% is low, and the diagnosis (too little data/capacity + a leaky benchmark) points straight at
  **v2: more data, more LoRA capacity, and decontamination.** → see [v2.md](v2.md).
