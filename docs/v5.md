# v5 — Targeted fixes: abstention, count-grounding, naming

**One line:** same 14B/MLX/LoRA recipe as v4, with the **data** upgraded to fix the three failure
modes v4's evaluation exposed. **(Training in progress — and it gave us a real crash/resume story,
below.)**

Read [v4.md](v4.md) first — v5 keeps v4's entire model/config/pipeline and changes **only the data**.
This is "data-centric" iteration: don't touch the knobs that work, fix the data that doesn't. Same 9
sections.

---

## 1. Memory math
Identical to v4: `Qwen2.5-14B-Instruct-MLX-8bit` — **8-bit ≈ 15 GB** frozen base, **~29 GB peak** while
training, on the 48 GB M4 Pro.

## 2. Mixed precision
Identical to v4: 8-bit base + small full-precision LoRA adapters via MLX.

## 3. Quantization
Identical **QLoRA-style** setup. See [v3 §3](v3.md) / [v4 §3](v4.md).

## 4. LoRA + gradient accumulation — **deliberately unchanged**
- Same **rank 24 / scale 10**, max_seq 1536, batch 4, `mask_prompt`, `grad_checkpoint`, **3300 iters
  (~2.5 epochs)**, cosine 5e-5→5e-6, 80-step warmup.
- **Why unchanged:** v4's training was clean (val 0.039, no overfit). Changing data *and* config at
  once makes any result impossible to attribute. So v5 changes **only the data** — a controlled
  experiment.

## 5. Checkpoints
- v4 used `save_every: 550`. After the crash below, the resumed run uses **`save_every: 200`** — finer
  checkpoints so a future crash loses ≤200 iters, and finer granularity for best-by-probe selection.
- *This version is the textbook argument for frequent checkpoints* — see §6.

## 6. resume_from_checkpoint — a real crash, recovered
This is the live illustration of why §5–6 matter:
- The laptop **powered off mid-run** at **iter ~975**. Because `save_every` was still 550, **only the
  iter-550 checkpoint had been saved** — iters 550→975 were lost.
- **The mlx-lm resume caveat (from [v3 §6](v3.md)) bit here:** `--resume-adapter-file` reloads **only
  the adapter weights — not the optimizer state or the step counter**. So resuming = load the iter-550
  weights and continue on a **fresh LR schedule** (re-warmup + a new cosine decay). (HF's
  `resume_from_checkpoint` would have restored optimizer + step seamlessly; mlx-lm does not.)
- **How we resumed cleanly:** load iter-550, set **iters 3300 → 2750** (the remaining iters to reach
  the planned ~3300 total) and the **cosine decay length → 2750** so the LR anneals fully over the
  resumed run; drop **save_every → 200**. Proof it worked: the resumed run's **iter-1 val loss (0.105)
  matched the pre-crash iter-550 val (0.104)** — the weights loaded correctly. Because the shipped
  checkpoint is chosen **by probe score across checkpoints** (not final loss), the one-time schedule
  discontinuity doesn't affect the final pick.

## 7. KB → training data — v4 pipeline + three data fixes
Same deterministic `build_dataset.py` (canonical layer, exhaustive 1-/2-/3-way lists with
list-then-count, paraphrase bank, decontamination), plus, each tied to a measured v4 failure:

1. **Abstention** *(v4 hallucinated companies in empty Glynn County)*. The KB covers only ~50 of
   Georgia's **159 counties**, so a new **`abstain_examples()`** generator adds "there are no companies
   located in *X* County" plus empty Category×Role combinations → the model learns to **say "none"
   instead of inventing a plausible supplier.** Boosted *modestly* (~11% of data) so it doesn't swing
   to *over*-refusing.
2. **Count-grounding** *(v4 was off-by-one: said 70 Tier 1, 11 OEM)*. For any count question whose
   answer is **≤ 50 companies**, the training answer now **enumerates the companies and then states the
   count** ("…there are 12 companies classified as OEM: <list>"), with the number forced to equal the
   list length — so the model **derives** the count instead of recalling it blind. (Huge sets like
   Tier 1 = 71 can't be listed → stay number-only; the residual off-by-one there is the pure-FT ceiling.)
3. **Naming fix** *(probe count confusion)*. v4 had a **role bucket** named "OEM Footprint" colliding
   with a **category** also named "OEM Footprint". v5 renames the role bucket to **"OEM Corporate
   Footprint"** so role-questions and category-questions stop bleeding into each other.

**Held out & measured:** 5 empty counties (Glynn, Echols, Quitman, Taliaferro, Webster) are
**reserved out of training** and added to the probe benchmark, so the abstention fix is *measured on
unseen counties*, not memorized. Probe set: **206 questions** (now 8 none-match probes). **Sizes:**
train **5,635** / valid 248 / test 50. All validation gates pass (0 leakage, count==listed,
refusal/abstention fraction in range, scorer ceiling 100%).

## 8. Results — *to be finalized after the run*
- **Targets:** probe overall **↑ from 85%**, **none-match 0/3 → ~100%**, count questions improved,
  company recall **held at ~97–100%**. The 14-question independent test should improve on abstention
  and small-set counts.
- **Honest expectation:** **multi-hop and comparative reasoning, and off-by-one on huge sets, will NOT
  fully resolve** — those are the structural limits of pure fine-tuning. (This doc's §8 will be filled
  with the measured numbers once training completes and `evaluate_mlx.py` + the probe scorer run.)

## 9. Documentation / are we going in the right direction?
- **Reproducibility / infra notes:** the project was moved **off iCloud Desktop to `~/dev/`** — iCloud
  was saturating the disk syncing multi-GB adapters and stalling jobs (a real "manage your hardware
  constraints" lesson). A **data-quality discovery**: the KB's 10 "Vehicle Assembly" rows are
  **column-misaligned** (Company/Address real, but OEM/Product belong to a different entity); the user
  chose to **leave it as-is** (match the literal KB / human gold) — documented so the next person
  doesn't re-discover it.
- **Direction:** yes — and this version makes the *limits* explicit. v5 is the strongest **pure
  fine-tuning** model; what it cannot do by itself — guarantee exact arithmetic and multi-hop joins
  over the table — is a known wall whose only real fix is the deferred **inference-time KB-compute /
  retrieval layer** (compute the answer deterministically at query time, then ground the model on it).
  Beyond v5, that hybrid is the next lever — not more epochs.
