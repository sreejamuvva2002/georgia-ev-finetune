# v5 — Targeted fixes: abstention, count-grounding, naming — explained

**One line:** same 14B/MLX/LoRA recipe as v4, but the data is upgraded to fix the three failure modes
v4's evaluation exposed. **(Training in progress as of writing.)**

Read [v4.md](v4.md) first — v5 keeps v4's entire model/config/pipeline and only changes the **data**.
This is "data-centric" iteration: don't touch the knobs that work, fix the data that doesn't.

---

## 1. Model, memory, config — deliberately unchanged
- Same `Qwen2.5-14B-Instruct-MLX-8bit` (8-bit ≈ 15 GB frozen base) + LoRA **rank 24 / scale 10**,
  max_seq 1536, batch 4, `mask_prompt`, `grad_checkpoint`, **3300 iters (~2.5 epochs)**, cosine
  5e-5→5e-6, save_every 550. Peak mem ~29 GB.
- We **reused the proven config** because v4's training was clean (val loss 0.039, no overfit). Changing
  data + config at once would make it impossible to attribute any change — so v5 changes **only data**.

## 2. The three data fixes (each tied to a measured v4 failure)
1. **Abstention** *(v4 hallucinated companies in empty Glynn County)*. The KB only covers ~50 of
   Georgia's **159 counties**, so v5 adds training examples for the other ~100: "there are no companies
   located in *X* County," plus empty Category×Role combinations. Result: the model learns to **say
   "none" instead of inventing a plausible supplier.** (Boosted *modestly* — ~11% of data — so it
   doesn't swing to *over*-refusing.)
2. **Count-grounding** *(v4 was off-by-one: said 70 Tier 1, 11 OEM)*. For any count question whose
   answer is ≤ 50 companies, the training answer now **enumerates the companies and then states the
   count** ("…there are 12 companies classified as OEM: <list>"). The number is forced to equal the
   list length, so the model derives the count instead of recalling it blind. (Huge sets like Tier 1 =
   71 can't be listed, so those stay number-only — the residual off-by-one there is the pure-FT ceiling.)
3. **Naming fix** *(probe count confusion)*. v4 had a **role bucket** named "OEM Footprint" colliding
   with a **category** also named "OEM Footprint". v5 renames the role bucket to **"OEM Corporate
   Footprint"** so role-questions and category-questions stop bleeding into each other.

## 3. How we turn the KB into training data (v5 = v4 + the above)
Same deterministic `build_dataset.py` pipeline as v4 (canonical conventions, exhaustive 1-/2-/3-way
lists with list-then-count, paraphrase bank, decontamination), plus:
- a new `abstain_examples()` generator (county + empty-cross "none" answers),
- count-grounding inside the count generators,
- the role-bucket rename.
- **Held-out & measured:** 5 empty counties (Glynn, Echols, Quitman, Taliaferro, Webster) are
  **reserved out of training** and added to the probe benchmark, so the abstention fix is *measured on
  unseen counties*, not just memorized. Probe set: **206 questions** (now 8 none-match probes).
- Sizes: train **5,635** / valid 248 / test 50. Validation gates all pass (0 leakage, count==listed,
  refusal/abstention fraction in range, scorer ceiling 100%).

## 4. An infrastructure note (your "documentation/reliability" themes)
- The project was moved **off iCloud Desktop** to `~/dev/` — iCloud was saturating the disk syncing the
  multi-GB adapters and stalling every job. (A real "manage your hardware constraints" lesson.)
- A **data-quality discovery** during v5 planning: the KB's 10 "Vehicle Assembly" rows are
  **column-misaligned** (Company/Address real, but OEM/Product belong to a *different* entity). The user
  chose to **leave it as-is** (match the literal KB / human gold) rather than fabricate a correction —
  so v5 does **not** alter those rows. Documented so the next person doesn't re-discover it.

## 5. Expected results (to be filled after training)
- Target: **probe overall ↑ from 85%**, **none-match 0/3 → ~100%**, count questions improved, company
  recall held at ~97–100%. The 14-question independent test should improve on abstention (Q4) and the
  small-set counts (Q7).
- Honest expectation: **multi-hop (H3) and comparative (H4) reasoning, and off-by-one on huge sets,
  will NOT fully resolve** — those are the structural limits of pure fine-tuning.

## 6. Are we going in the right direction?
Yes — and this version makes the *limits* explicit. The accuracy arc (Base 4% → v1 26% → v2 54% →
v3 46% → v4 54% + 97.5% recall → v5 targeting the last fixable gaps) shows the **data-centric LoRA
approach is sound and converging**. What it *cannot* do by itself — guarantee exact arithmetic and
multi-hop joins over the table — is a known wall; the only way through it is the deferred **inference-
time KB-compute / retrieval layer** (compute the answer deterministically at query time, ground the
model on it). v5 is the strongest *pure-fine-tuning* model; beyond it, the next real lever is that
hybrid, not more epochs.
