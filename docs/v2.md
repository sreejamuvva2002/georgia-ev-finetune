# v2 — Bigger data + capacity + decontamination (7B)

**One line:** same 7B, but more LoRA capacity, eval-aligned data, and a leakage guard. Result:
**v1 26% → v2 54%** on the 50 held-out human Q&A. v2 was the **shipped 7B model**.

Read [v1.md](v1.md) for the full concept explanations; this doc keeps the same 9 sections but
focuses on **what changed and why it doubled accuracy**.

---

## 1. Memory math
Unchanged from v1: same `Qwen/Qwen2.5-Coder-7B-Instruct` (7.6B). **BF16 ≈ 15 GB** of weights on the
48 GB M4 Pro; the rest of the budget goes to activations/gradients/optimizer. The one capacity bump
(LoRA rank, §4) adds only a small amount of trainable params, so it stays comfortably in budget.

## 2. Mixed precision
Unchanged: **BF16 on MPS** (FP32 range, half the memory). See [v1 §2](v1.md).

## 3. Quantization
Unchanged: **no base quantization on the Mac** (bf16 fits). 4-bit QLoRA remains reserved for the
CUDA/vLLM deploy box via `train_qlora.py`'s auto-switch.

## 4. LoRA + gradient accumulation — **capacity up**
- **r=16 → r=32, α=32 → α=64.** This doubles the adapter to **80.7M trainable params** (~2× v1).
  - Intuition: **rank = how much new behaviour the adapters can store.** The KB has many exact facts
    (193 companies × many attributes + aggregates) to bind, so more rank = more room.
- **`max_seq_length = 1,536`** so longer list-style answers aren't truncated.
- Everything else identical to v1: micro-batch 1 × **grad-accum 16** (effective batch 16), **lr 2e-4
  cosine, 3 epochs**, bf16.

## 5. Checkpoints
Same policy as v1 — **checkpoint every 100 steps** (snapshotting weights + optimizer + scheduler +
step), kept from v1's disk-crash lesson.

## 6. resume_from_checkpoint
Same HF `resume_from_checkpoint=True` path as v1 (seamless resume with optimizer/scheduler/step
restored). One eval-time gotcha this version: a load-time crash (only the HIGH MPS watermark set, not
LOW) silently left **v1's** answer file in place, so the first score *looked* like "v2 = 26%". We
caught it by diffing v2-vs-v1 answers (0/50 differed → suspicious), set both env vars, and re-ran. —
*Lesson: a suspiciously unchanged result is a bug signal.*

## 7. KB → training data — three changes
Same deterministic pandas pipeline as v1 (**nothing invented**), plus:

1. **Decontamination (integrity).** v1 had mirrored 8 eval *question* phrasings into training. v2
   adds a **leakage guard** in `add()`: any generated question that matches a held-out test question
   — **exact-normalized OR ≥0.85 token-Jaccard** near-duplicate — is **dropped**. It removed **73**
   leaky generations. Verified **0** test-question and **0** gold-answer leakage → all 50 genuinely
   held out. *This is why v2's 54% is both higher and more trustworthy than v1's 26%.*
2. **Eval-aligned data (`agg_examples_v2`).** Re-emit the high-value query shapes the eval is built
   from (battery roles, thermal management, Tier-2/3 filters, county-employment sums/argmax, OEM
   linkage, dual-platform, site-selection) with **3–4 paraphrases each**, oversampled ×3.
   - **Why paraphrases:** so the model binds the underlying *fact* (the real list/number), not one
     question string. This is the single biggest data lever.
3. **Sizes:** train **2,535** / valid 196 / test 50.

## 8. Results
| Model | Accuracy (50 held-out) | Mean score |
|---|---|---|
| Base 7B | 4% (2/50) | 0.201 |
| v1 (r=16) | 26% (13/50) | 0.363 |
| **v2 (r=32, shipped)** | **54% (27/50)** | **0.584** |

- Training: **train loss 0.288, eval loss 0.195**, token-acc 0.955, **~2 h 35 m**.
- Per category (v1→v2): Mapping 4→8/16 · Discovery 3→5/11 · Trends 2→5/11 · Risk 2→5/7 · Site 2→4/5.
- **What the extra capacity + paraphrases fixed:** facts v1 fabricated became correct — highest-
  employment county (Troup→Gwinnett), Tier-1 county totals, battery/Tier lists, OEM linkage.

## 9. Documentation / are we going in the right direction?
- **The honest ceiling found here:** v2 exposed the **structural limit of pure fine-tuning** — exact
  table-wide aggregation and exact long-list enumeration under tight filters still slip (the model
  *recalls* facts parametrically; it can't reliably *compute*). We told the user plainly: pure FT
  realistically tops out **~55–65%**; the robust route to 75–80% is **inference-time
  retrieval/compute** (inject exact KB rows at query time). **The user chose to keep pushing pure
  fine-tuning.**
- **Direction:** strongly yes — 26%→54% from *better data + capacity + integrity* validates the
  data-centric approach, and decontamination makes the number trustworthy. The open question it
  raised — *can a bigger model break the ceiling?* — led to the **14B (v3).** → see [v3.md](v3.md).
