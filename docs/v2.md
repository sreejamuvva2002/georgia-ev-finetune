# v2 — Bigger data + capacity + decontamination (7B) — explained

**One line:** same 7B, but more LoRA capacity, eval-aligned data, and a leakage guard. Result:
v1 26% → **v2 54%** on the 50 held-out human Q&A. v2 was the **shipped 7B model**.

Read [v1.md](v1.md) first for the concept basics (memory math, LoRA, gradient accumulation,
checkpoints). This doc focuses on **what changed** and **why it doubled accuracy**.

---

## 1. Model & memory — unchanged
- Same base `Qwen/Qwen2.5-Coder-7B-Instruct`, same **bf16 on MPS (~15 GB)**, same no-quantization path
  on the Mac (4-bit QLoRA still reserved for the CUDA deploy box).

## 2. What changed #1 — more LoRA capacity
- **r=16 → r=32, α=32 → α=64.** This doubles the adapter size to **80.7M trainable params** (~2× v1).
- Intuition: rank = how much "new behaviour" the adapters can store. The KB has lots of exact facts
  (193 companies × many attributes + aggregates) to memorize, so more rank = more room to bind them.
- `max_seq_length = 1,536` so longer list-style answers aren't truncated.

## 3. What changed #2 — decontamination (integrity)
- v1's generator had accidentally mirrored 8 eval *question* phrasings into training. v2 added a
  **leakage guard**: any generated question that matches a held-out test question (exact-normalized
  **or** ≥0.85 token-Jaccard near-duplicate) is **dropped**. It dropped 73 leaky generations.
- Verified **0** test-question and **0** gold-answer leakage → all 50 are genuinely held out. This is
  why v2's 54% is both *higher and more trustworthy* than v1's 26%.

## 4. What changed #3 — eval-aligned data ("data-centric")
- New `agg_examples_v2()` re-emits the high-value query shapes the eval is built from (battery roles,
  thermal management, Tier-2/3 filters, county-employment sums/argmax, OEM linkage, dual-platform,
  site-selection) with **3–4 paraphrases each**, oversampled ×3.
- **Why paraphrases:** so the model binds the underlying *fact* (the real company list/number), not a
  single question string. This is the single biggest data lever.
- Sizes: train **2,535** / valid 196 / test 50.

## 5. Training mechanics (same recipe as v1)
- LoRA on all 7 projections, **lr 2e-4 cosine, 3 epochs**, **effective batch 16** (micro-batch 1 ×
  **gradient accumulation 16**), bf16.
- **Checkpointing every 100 steps + auto-resume** (kept from v1's disk-crash lesson).
- Training: **train loss 0.288, eval loss 0.195**, token-acc 0.955, ~**2 h 35 m**.

## 6. KB → training data (same deterministic pipeline)
Identical philosophy to v1 — **all facts computed from the KB in pandas, nothing invented** — plus the
v2 additions above (decontam guard + eval-aligned paraphrased aggregates). The 50 human Q&A remain the
untrained held-out test.

## 7. Results
| Model | Accuracy (50 held-out) | Mean score |
|---|---|---|
| Base 7B | 4% (2/50) | 0.201 |
| v1 (r=16) | 26% (13/50) | 0.363 |
| **v2 (r=32, shipped)** | **54% (27/50)** | **0.584** |

Per category (v1→v2): Mapping 4→8/16 · Discovery 3→5/11 · Trends 2→5/11 · Risk 2→5/7 · Site 2→4/5.

**What the extra capacity + paraphrases fixed:** facts v1 fabricated became correct — highest-
employment county (Troup→Gwinnett), Tier-1 county totals, battery/Tier lists, OEM linkage.

## 8. Honest ceiling found here
v2 also exposed the **structural limit of pure fine-tuning**: exact table-wide aggregation and exact
long-list enumeration under tight filters still slip (the model recalls facts parametrically; it
can't reliably *compute*). We told the user plainly: pure FT realistically tops out ~55–65%; the
robust route to 75–80% is **inference-time retrieval/compute** (inject exact KB rows at query time).
The user chose to keep pushing pure fine-tuning.

## 9. Are we going in the right direction?
Yes — 26%→54% from *better data + capacity + integrity* validates the data-centric approach, and the
decontamination makes the number trustworthy. The open question it raised — "can a bigger model break
the ceiling?" — led to the 14B (v3). → see [v3.md](v3.md).
