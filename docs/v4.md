# v4 вЂ” No-compromise 14B: fix the data, measure honestly

**One line:** same 14B/MLX as v3, but with the data bugs fixed, more LoRA capacity, and a proper
held-out benchmark. Result: **54%** on the 50 (ties the best) **and 97.5% company recall** on a new
201-question probe set вЂ” the "missing companies" problem is gone, and refusals work again.

Read [v3.md](v3.md) for the MLX/quantization basics. Same 9 sections; the theme here is **getting the
data and the measurement right** вЂ” the two things that actually moved the needle.

## 4. LoRA + gradient accumulation вЂ” **more capacity**
- **rank 16 в†’ 24** (**103M** trainable params). Higher rank в‡’ more room to memorize the exhaustive
  membership lists v4 generates. **Scale kept at 10** (raising it caused v3's divergence вЂ” left alone).
- **max_seq_length 1024 в†’ 1536** (list-then-count answers are longer), batch 4, `mask_prompt`,
  `grad_checkpoint`.
- **3300 iters (~2.5 epochs)**, cosine **5e-5 в†’ 5e-6**, **80-step warmup**. Still **batch 4, no grad
  accumulation** (MLX fits it directly).

## 5. Checkpoints
- **`save_every: 550` в†’ 6 checkpoints** across the run. Crucially, we **don't assume the last
  checkpoint is best** вЂ” see В§8/В§9.
- The run finished in a few hours вЂ” **cooler than v3's 12.5 h** (no competing processes; see v3 В§5's
  swap-thrash lesson, applied here).

## 6. resume_from_checkpoint
Same mlx-lm `--resume-adapter-file` mechanism as v3 (weights-only resume). v4's run completed cleanly
without needing a resume вЂ” but the checkpoint discipline set up the key v4 practice: **select the
shipped adapter by held-out probe score, not by final loss** (because memorization can keep lowering
train loss past the point of best generalization).

## 7. KB в†’ training data вЂ” the three fixes + the full pipeline
Everything is **deterministic pandas вЂ” no model invents a fact** (`build_dataset.py`).

**The three fixes that made v4 work:**
1. **One counting convention.** Verified against the raw gold that the human answers count **unique
   companies** (e.g. "4 Power-Electronics companies" counts ZF once despite its 3 rows). v4
   standardizes on **unique companies everywhere** вЂ” generators, probe golds, and scorer.
2. **Role normalization (`role_tags`).** The KB's `EV Supply Chain Role` has 8 clean roles **+ 26
   free-text one-offs**. v4 buckets them with **substring rules** (anything containing "thermal" в†’
   Thermal Management, "harness" в†’ Wiring Harness, вЂ¦) so the one-offs stop being dropped. Plus an
   **OEM splitter (`oem_tags`)**: "Hyundai Kia Rivian" feeds the Hyundai *and* Kia *and* Rivian
   supplier lists вЂ” so **Rivian now returns all 6 companies** (v3 returned 2).
3. **Restored refusals.** v3 over-trained "always answer from the KB" and lost the ability to say
   "not in the KB". v4 expands refusals + adds explicit **"0 results"** answers and rebalances their
   oversampling.

**The pipeline (the generators in `build_dataset.py`):**
- **Canonical layer:** `uniq()` (unique companies), `role_tags()` (RoleNorm buckets), `oem_tags()`
  (OEM splitter), `category_norm()` (merge duplicate "OEM (Footprint)" labels), and **name
  canonicalization** (2 spelling-duplicate companies, e.g. "Trenton Pressing" / "вЂ¦Inc." в†’ 193 names
  are really **191 companies**).
- **`company_examples`** вЂ” per-company facts for all 191 companies, every attribute.
- **`list_examples`** вЂ” every 1-way filter (Category, RoleNorm, County, City, Industry, Relevance,
  OEM), each emitting the **full unique membership + a count equal to the list length**
  ("**list-then-count**" вЂ” the count is grounded in the shown list, not recalled blind).
- **`cross_examples`** вЂ” key 2-way / 3-way crosses (e.g. CategoryГ—RoleГ—Relevance).
- **`agg_examples`** вЂ” counts/distributions, employment sums, argmax, top-K, single-source roles,
  gap-counties.
- **`keyword_examples`** вЂ” product-keyword search; **`refusal_examples`** вЂ” refusals + "0 results".
- **Paraphrase bank** (`vary()` over hand-authored templates) вЂ” many phrasings per fact, offline.
- **Decontamination:** drop anything overlapping the 50 human Q&A **and** a new **201-question probe
  benchmark** (0 leakage verified).
- **Sizes:** train **5,289** / valid 229 / test 50.

**Measuring honestly вЂ” the probe benchmark.** Because the 50-question gold is **internally
inconsistent** (Q3 vs Q42 ask the same thing, gold says 5 vs 4), it can't be the target. v4's
**primary metric** is the **201-probe** benchmark with a scorer **verified to a 100% ceiling on
perfect answers** вЂ” so any shortfall is a *real* model error, not scorer noise. Headline = **company
recall** (the "no missing companies" gate).

## 8. Results
| Metric | v4 |
|---|---|
| 50 human Q&A (score в‰Ґ 0.6) | **54%** (27/50), mean 0.598 вЂ” ties v2, beats v3 |
| Probe **company recall** | **97.5%** (missing-company rate 2.5%) вЂ” vs v3's Rivian 2/6 |
| Refusals | **100%** (v3 regression fixed) |
| Probe overall (strict) | 85.1% |
| Weak spots | exact **count** 58% (off-by-one); **none-match** 0/3 |

- **Loss:** final **val 0.039**; the **best checkpoint by val loss was iter-2200 @ 0.031** вЂ” *not* the
  final iter вЂ” which is exactly why we select by held-out score, not final loss.
- On an independent 14-question test it scored **7/14**, failing on **abstention** (invented companies
  in an empty county), **exact counting** (off-by-one; over-counting AND-conditions), and
  **multi-hop/comparative** reasoning.

## 9. Documentation / are we going in the right direction?
- **Documented:** the canonical conventions, the probe scorer's verified ceiling, and a clean failure
  taxonomy splitting **fixable** (abstention, count-grounding) from **structural** (exact aggregation,
  multi-hop вЂ” the pure-FT ceiling).
- **Direction:** strongly yes for the stated goal (**company coverage / no missing companies**): 97.5%
  recall, fixed refusals, best human-eval score. The fixable gaps define **v5**; the structural ones
  ultimately need an inference-time compute layer. в†’ see [v5.md](v5.md).
