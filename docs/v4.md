# v4 — No-compromise 14B: fix the data, measure honestly — explained

**One line:** same 14B/MLX as v3, but with the data bugs fixed, more LoRA capacity, and a proper
held-out benchmark. Result: **54%** on the 50 (ties the best) **and 97.5% company recall** on a new
201-question probe set — the "missing companies" problem is gone, and refusals work again.

Read [v3.md](v3.md) for the MLX/quantization basics. This doc is about **getting the data and the
measurement right** — the two things that actually moved the needle.

---

## 1. Model, memory, LoRA config
- Same `Qwen2.5-14B-Instruct-MLX-8bit` (8-bit ≈ 15 GB frozen base) + LoRA — the **QLoRA-style** setup.
- **Capacity up: rank 16 → 24** (103M trainable params). Higher rank ⇒ more room to memorize the
  exhaustive membership lists. **Scale kept at 10** (raising it caused v3's divergence — left alone).
- **max_seq_length 1024 → 1536**, batch 4, `mask_prompt`, `grad_checkpoint`.
- **3300 iters (~2.5 epochs)**, cosine 5e-5→5e-6, 80-step warmup. **save_every 550 → 6 checkpoints.**
- Peak memory ~29 GB; the run finished in a few hours (cooler than v3's 12.5 h).

## 2. The three data fixes that made v4 work
1. **One counting convention.** We verified against the raw gold that the human answers count **unique
   companies** (e.g. "4 Power-Electronics companies" counts ZF once despite its 3 rows). v4 standardizes
   on **unique companies everywhere** — generators, probe golds, and scorer. *(The session logs had
   claimed the opposite; checking the raw data corrected it.)*
2. **Role normalization (`RoleNorm`).** The KB's `EV Supply Chain Role` has 8 clean roles **+ 26 free-
   text one-offs**. v4 buckets them with substring rules (anything containing "thermal" → Thermal
   Management, "harness" → Wiring Harness, etc.) so the model stops dropping the one-offs. Plus an
   **OEM splitter**: "Hyundai Kia Rivian" feeds the Hyundai *and* Kia *and* Rivian supplier lists — so
   **Rivian now returns all 6 companies** (v3 returned 2).
3. **Restored refusals.** v3 over-trained "always answer from the KB" and lost the ability to say "not
   in the KB". v4 expands refusals + adds explicit "0 results" answers and balances their oversampling.

## 3. How we turn the KB into training data (the full v4 pipeline)
Everything is **deterministic pandas — no model invents a fact**. `build_dataset.py`:
- **Canonical layer:** `uniq()` (unique companies), `role_tags()` (the RoleNorm buckets),
  `oem_tags()` (OEM splitter), `CategoryNorm` (merge duplicate "OEM (Footprint)" labels), and
  **company-name canonicalization** (the KB had 2 spelling-duplicate companies, e.g. "Trenton Pressing"
  / "Trenton Pressing Inc." → 193 names are really **191 companies**).
- **Generators:**
  - *Per-company facts* (all 191 companies, every attribute).
  - *Exhaustive lists* — every 1-way filter (Category, RoleNorm, County, City, Industry, Relevance,
    OEM) **and** key 2-way / 3-way crosses, each emitting the **full unique membership + a count that
    equals the list length** ("list-then-count" — so the count is grounded in the shown list).
  - *Aggregates* — counts/distributions, employment sums, argmax, top-K, single-source roles,
    gap-counties, etc.
  - *Product-keyword search*, *refusals*, *"0 results"*.
- **Paraphrase bank:** many natural phrasings per fact (offline, hand-authored) so the model binds the
  fact, not one string.
- **Held-out:** the 50 human Q&A **and** a new **201-question probe benchmark** (KB-computed gold,
  *reserved* phrasings) — training is **decontaminated against both** (0 leakage verified).
- Sizes: train **5,289** / valid 229 / test 50.

## 4. Measuring honestly — the probe benchmark
Because the 50-question gold is **internally inconsistent** (Q3 vs Q42 ask the same thing, gold says
5 vs 4), it can't be the target. v4's **primary metric** is the 201-probe benchmark with a scorer
**verified to a 100% ceiling on perfect answers** — so any shortfall is a *real* model error, never
scorer noise. Headline metric = **company recall** (the "no missing companies" gate).

## 5. Results
| Metric | v4 |
|---|---|
| 50 human Q&A (score ≥ 0.6) | **54%** (27/50), mean 0.598 — ties v2, beats v3 |
| Probe **company recall** | **97.5%** (missing-company rate 2.5%) — vs v3's Rivian 2/6 |
| Refusals | **100%** (v3 regression fixed) |
| Probe overall (strict) | 85.1% |
| weak spots | exact **count** 58% (off-by-one), **none-match** 0/3 |

## 6. What v4 still gets wrong (and why)
On an independent 14-question test it scored 7/14, failing on: **abstention** (it invented companies
in an empty county instead of saying "none"), **exact counting** (off-by-one; over-counting AND-
conditions, e.g. "Tier 1 ∧ EV=Yes" → said 7, truth 1), and **multi-hop/comparative reasoning** (the
pure-FT ceiling). The first two are **fixable data gaps** → v5.

## 7. Are we going in the right direction?
Strongly, for the user's stated goal (**company coverage / no missing companies**): v4 delivers 97.5%
recall and fixes v3's regressions while matching the best human-eval score. The residual gaps split
into **fixable** (abstention, count-grounding — v5) and **structural** (exact aggregation, multi-hop —
only an inference-time compute layer fully fixes those). → see [v5.md](v5.md).
