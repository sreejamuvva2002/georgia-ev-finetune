# v3 — Going bigger: 14B on MLX (QLoRA-style)

**One line:** swap the 7B for a **14B**, trained with Apple's MLX because the 14B we had was already
**8-bit quantized**. Result: **46%** on the 50 held-out — *on par with v2's 54%, not a clear win* —
but it exposed the real fixes for v4.

Read [v2.md](v2.md) for context. Same 9 sections; the new themes here are the **toolchain switch
(MLX), the quantization story, and why more parameters didn't automatically beat v2.**

---

## 1. Memory math
- **Base model:** `Qwen2.5-14B-Instruct-MLX-8bit` — **14.1B** parameters.
- For 14.1B params:
  - **BF16** (2 bytes) → ~28 GB — tight on 48 GB once activations/optimizer are added.
  - **8-bit** (1 byte) → **~15 GB** ← what the MLX build gives us; fits comfortably.
- So the jump from 7B→14B was only affordable *because* the base is quantized (see §3).

## 2. Mixed precision
MLX handles the compute precision internally; the meaningful precision lever here is that the **base
weights are stored 8-bit** (not bf16/fp16). The trainable LoRA adapters are small and full-precision.

## 3. Quantization — the QLoRA idea, on Apple silicon
- The base is **genuinely 8-bit quantized** (config: `"quantization": {"bits": 8, "group_size": 64}`).
- This is the **QLoRA pattern**: a **quantized, frozen base** + **trained LoRA adapters** on top.
  Here the quantization is **MLX 8-bit (Apple)** rather than bitsandbytes 4-bit (CUDA), but the
  principle is identical — shrink the frozen base so a big model fits, train only small adapters.
- **Toolchain consequence:** HF/PEFT **cannot train MLX-quantized weights**, so for the 14B we
  switched to Apple's **`mlx-lm`** (`mlx_lm.lora`) instead of the HF/PEFT path used for the 7B.
  Trade-off accepted: the MLX adapter is **Mac-only** (not a portable HF adapter). The training
  *data* is model-agnostic chat JSONL, so a Linux/HF rebuild is always possible later.

## 4. LoRA + gradient accumulation
- **rank 16, scale 10**, dropout 0.05, LoRA on all **48 layers** (q/k/v/o/gate/up/down).
- **`mask_prompt: true`** = compute loss **only on the assistant's tokens** (completion-only) — don't
  train on the system+user prompt.
- **`grad_checkpoint: true`** = recompute activations during the backward pass to save memory (trades
  compute for memory — important on a memory-bound Mac).
- **batch_size 4 — no gradient accumulation needed here.** MLX fit 4 sequences directly (unlike the
  7B's 1×16). **max_seq_length 1024**, **1200 iters (~1.5 epochs)**, cosine **5e-5 → 5e-6** with a
  **60-step warmup**.

**A real divergence (why warmup + a sane LR matter):** the **first config diverged** — lr 1e-4 +
scale 20 + **no warmup** → loss went **2.95 → 6.5 by iter 75** (the LoRA updates were too large and
the model fell apart). Fix: **halve the scale (20→10), lower the peak LR, add warmup**, and run a
**90-iter "stability probe"** to confirm clean descent *before* committing to the long run. This
lesson is baked into every later version.

## 5. Checkpoints
- **Checkpoint every 300 iters.** An **iter-300 fallback** adapter was preserved
  (`adapters/georgia_ev_14b_mlx_iter300_fallback.safetensors`).
- **Operational lesson (memory):** a leftover gradio demo + chat REPL (each ~15 GB) ran alongside
  training → **39 GB swap thrash**, dragging it to 40–90 s/iter. Killing them → clean ~24 GB peak.
  *Don't run other big-model processes during training.*
- **Hardware reality:** ~10 s/iter cool → **~40 s/iter thermally throttled**; the full 1200-iter run
  took **~12.5 h** (vs ~1–3 h on a cloud A100). On this Mac, the GPU/thermals — not RAM — is the limit.

## 6. resume_from_checkpoint
- mlx-lm resumes with **`--resume-adapter-file <adapters.safetensors>`**. **Important difference from
  HF:** mlx-lm's resume reloads **only the adapter weights** — **not** the optimizer state or the step
  counter — so the LR schedule restarts. (HF's `resume_from_checkpoint` restores optimizer + scheduler
  + step and continues seamlessly.) This distinction matters a lot for v5's crash recovery — see
  [v5 §6](v5.md).

## 7. KB → training data
- Added **`agg_examples_v3()`**: exhaustive **Category×Role / Category×Relevance / Role×Relevance /
  Category×OEM** filtered lists (count == list by construction), distribution cross-tabs, argmax,
  sums — paraphrase-rich. train **2,535 → 3,206**; valid 221; test 50; **0 exact leakage** verified.
- **Two bugs introduced here (the real reason v3 didn't beat v2):**
  1. **Count convention.** v3 counted **unique companies** (`drop_duplicates`) while the probe golds
     and human gold were (at the time) treated as **rows**. KB = 205 rows / 193 unique (Tier 1 = 77
     rows / 71 unique). The 14B learned *unique* counts and scored "wrong" against row-count gold.
  2. **Under-listing.** v3's role filter matched **exact role strings**, silently dropping the KB's
     **26 free-text one-off roles** → the model under-listed (e.g. "Rivian suppliers" returned 2 of 6).
  - Both are **data/convention bugs, not model-capacity limits** — and both are fixed in v4.

## 8. Results
- **Train loss 0.056, val loss 0.158** — notably more overfit than v4 would be.
- **46% (23/50)**, mean 0.605 — higher *mean* than v2 but lower *pass-rate* (14 near-misses). It did
  solve the **superlative/argmax** class v2 couldn't (e.g. "Gwinnett's highest employer → WIKA
  250,000").

## 9. Documentation / are we going in the right direction?
- **Documented for the next pass:** the two bugs above, the divergence recipe, the swap-thrash
  incident, and the KB normalization issues (duplicate `OEM (Footprint)` labels; role free-text
  one-offs; employment mixing global vs site headcounts) — so v4 wouldn't re-discover them.
- **Direction:** partly. The sharp lesson — **more parameters did NOT automatically help; clean,
  consistent data did.** v3's value was diagnostic: it pinpointed the count-convention and
  role-normalization bugs that, once fixed, drove v4. → see [v4.md](v4.md).
