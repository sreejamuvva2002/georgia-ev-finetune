# v3 — Going bigger: 14B on MLX (QLoRA-style) — explained

**One line:** swap the 7B for a **14B**, trained on Apple's MLX because the 14B we had was already
**8-bit quantized**. Result: **46%** on the 50 held-out — *on par with v2's 54%, not a clear win* —
but it exposed the real fixes for v4.

Read [v2.md](v2.md) for context. This doc explains the **toolchain switch (MLX), the quantization
story, and why more parameters didn't automatically beat v2.**

---

## 1. Model & memory — the quantization story
- **Base model:** `Qwen2.5-14B-Instruct-MLX-8bit` (14.1B params), **genuinely 8-bit quantized**
  (`bits:8, group_size:64`).
- **Memory math:**
  - BF16 would be 14B × 2 = **~28 GB** (tight on 48 GB once you add activations/optimizer).
  - **8-bit = 14B × 1 = ~15 GB** ← what the MLX build gives us, fits comfortably.
- **This is the QLoRA idea** from your notes: a **quantized, frozen base** + **trained LoRA adapters**
  on top. Here the quantization is **MLX 8-bit** (Apple) rather than bitsandbytes 4-bit (CUDA), but the
  principle is identical — shrink the frozen base so a big model fits, train only small adapters.

## 2. Why MLX (toolchain switch)
- HF/PEFT **cannot train MLX-quantized weights**. So for the 14B we used Apple's **`mlx-lm`**
  (`mlx_lm.lora`) instead of the HF/PEFT path used for the 7B.
- Trade-off the user accepted: MLX is **Mac-only** (the adapter isn't a portable HF adapter). The
  training *data* is model-agnostic chat JSONL, so a Linux/HF rebuild is always possible later.

## 3. LoRA + training config
- **rank 16, scale 10**, dropout 0.05, LoRA on all **48 layers** (q/k/v/o/gate/up/down).
- **`mask_prompt: true`** = compute loss only on the assistant's tokens (don't train on the
  system+user prompt) — completion-only loss.
- **`grad_checkpoint: true`** = recompute activations in the backward pass to save memory (trades
  compute for memory — important on a memory-bound Mac).
- **batch_size 4** (MLX fit 4 sequences directly — no gradient accumulation needed here, unlike the
  7B's 1×16), **max_seq_length 1024**, **1200 iters (~1.5 epochs)**.
- LR: **cosine 5e-5 → 5e-6 with a 60-step warmup.**

## 4. A real divergence (why warmup + a sane LR matter)
- The **first config diverged**: lr 1e-4 + scale 20 + no warmup → loss went **2.95 → 6.5 by iter 75**
  (the LoRA updates were too large, the model fell apart).
- Fix: **halve the scale (20→10), lower the peak LR, add warmup.** A 90-iter "stability probe"
  confirmed clean descent *before* committing to the long run. **Lesson baked into all later versions.**

## 5. Operational lessons (your notes in action)
- **Memory incident:** a leftover gradio demo + chat REPL (each ~15 GB) ran alongside training →
  **39 GB swap thrash**, dragging it to 40–90 s/iter. Killing them → clean ~24 GB peak. *Don't run
  other big-model processes during training.*
- **Hardware reality:** ~10 s/iter cool → **~40 s/iter thermally throttled**; the full 1200-iter run
  took **~12.5 h**. This is the practical ceiling of a Mac vs ~1–3 h on a cloud A100.
- **Checkpoints** every 300 iters; an iter-300 fallback adapter was preserved.

## 6. KB → training data (expanded, but with two bugs)
- Added `agg_examples_v3()`: exhaustive **Category×Role / Category×Relevance / Category×OEM** filtered
  lists, distributions, argmax, sums. train **2535 → 3206**.
- **Bug #1 (count convention):** v3 counted **unique companies** while the probe golds and human gold
  (we later found) were inconsistent — this mis-scored v3.
- **Bug #2 (under-listing):** v3's role filter used **exact role strings**, silently dropping the KB's
  **26 free-text one-off roles** → the model under-listed (e.g. "Rivian suppliers" returned 2 of 6).
- These two bugs — not the model size — are why v3 didn't beat v2. **All fixed in v4.**

## 7. Results
- Train loss **0.056**, val loss **0.158** (notably more overfit than v4 would be).
- **46% (23/50)**, mean 0.605 — higher mean than v2 but lower pass-rate (14 near-misses). It *did*
  solve the superlative/argmax class v2 couldn't (e.g. "Gwinnett's highest employer → WIKA 250,000").

## 8. Are we going in the right direction?
Partly. The lesson was sharp: **more parameters did not automatically help — clean, consistent data
did.** v3's value was diagnostic: it pinpointed the count-convention and role-normalization bugs that,
once fixed, drove v4. → see [v4.md](v4.md).
